4.5.2 Logika Data Bertingkat (Multi-condition)

Di aplikasi kompleks, keputusan tidak hanya ditentukan oleh satu data. Aplikasi sering memproses lebih dari satu input sebelum menentukan hasil. :contentReference[oaicite:6]{index=6}

Contoh gampang:
‚ÄúDiskon‚Äù tidak hanya cek umur, tapi juga status pelajar. Jadi hasil dipilih berdasarkan kombinasi data, bukan cuma ‚Äúya/tidak‚Äù. :contentReference[oaicite:7]{index=7}
Aturan 1
Cek kondisi cepat dulu
Aturan 2
Kalau lolos, lanjut cek berikutnya
Aturan 3
Kalau gagal, tampilkan pesan

4.5.3 State & Status dalam Alur Aplikasi

Aplikasi tidak selalu berjalan satu arah dari awal ke akhir. Dalam penggunaan sehari-hari, aplikasi punya state/status yang bisa berubah. :contentReference[oaicite:9]{index=9}

Contoh state:
  • Belum menerima input
  • Pengguna sedang mengisi data
  • Data sedang diproses
  • Hasil sudah ditampilkan
  • Terjadi kesalahan input
Ini disebut di modul sebagai contoh state aplikasi. :contentReference[oaicite:10]{index=10}
Contoh situasi nyata:
Tombol ditekan sebelum data dimasukkan ‚Üí aplikasi harus merespons dengan benar (kasih pesan, jangan lanjut). Atau data diubah setelah hasil ditampilkan ‚Üí aplikasi harus update. :contentReference[oaicite:11]{index=11}

Simulasi State: klik yang benar

Kamu punya TextBox umur dan tombol ‚ÄúCek‚Äù. Kamu klik tombol saat TextBox kosong. Apa yang harus terjadi?

Kenapa?
Karena state ‚Äúbelum input‚Äù harus punya respon yang jelas. Ini yang membuat aplikasi terasa ‚Äúbeneran‚Äù dipakai orang.

4.5.4 Validasi Data & Penanganan Kesalahan

Salah satu penyebab utama aplikasi bermasalah adalah data tidak valid. Maka sebelum memproses data, aplikasi perlu melakukan validasi. :contentReference[oaicite:12]{index=12}

Validasi memastikan:
  • Data tidak kosong
  • Tipe data sesuai
  • Nilai masih masuk akal
Jika tidak divalidasi:
  • Hasil keliru
  • Error logika
  • Pengguna bingung
Ini disebut di modul. :contentReference[oaicite:13]{index=13}
Kunci penting:
Kalau data tidak valid, aplikasi jangan lanjut proses. Tampilkan pesan error / minta perbaiki input. :contentReference[oaicite:14]{index=14}

Kenapa penting di App Inventor?

Di App Inventor, blok logika bisa tetap berjalan meskipun TextBox kosong atau input salah. Jadi validasi itu wajib. :contentReference[oaicite:15]{index=15}

Contoh kondisi yang sering terjadi:
  • TextBox masih kosong
  • Input bukan angka
  • Nilai tidak sesuai aturan aplikasi
Trik sederhana di App Inventor:
Semua validasi biasanya ditaruh di when Button.Click pakai if‚Äìthen‚Äìelse, sebelum proses utama.

Jenis validasi (yang sering dipakai)

Modul menyebut 3 jenis validasi yang umum di App Inventor. :contentReference[oaicite:16]{index=16}

1) Input kosong
Cek apakah pengguna sudah memasukkan data sebelum tombol ditekan. :contentReference[oaicite:17]{index=17}
2) Tipe data
Pastikan input sesuai jenis data yang diharapkan (misalnya angka untuk perhitungan). :contentReference[oaicite:18]{index=18}
3) Rentang nilai
Pastikan nilai masuk akal (misalnya umur tidak negatif). :contentReference[oaicite:19]{index=19}
Validasi ini biasanya dilakukan menggunakan if‚Äìthen‚Äìelse sebelum proses utama dijalankan. :contentReference[oaicite:20]{index=20}

Manfaat validasi (biar aplikasi ‚Äúpro‚Äù)

Dengan menerapkan validasi, aplikasi jadi: lebih tahan kesalahan, hasil lebih akurat, terasa profesional, dan mudah dikembangkan ke fitur yang lebih kompleks. :contentReference[oaicite:21]{index=21}

Tanpa validasi
  • Sering salah hitung
  • Pesan tidak jelas
  • Pengguna bingung
Dengan validasi
  • Input salah ketahuan cepat
  • Pesan error jelas
  • Flowchart & blok lebih rapi
